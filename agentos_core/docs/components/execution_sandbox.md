# Execution Sandbox — Internal Component

## Responsibility

The **Execution Sandbox** (implemented in `sandbox/manager.py` and `sandbox/worker.py`) provides a secure, isolated environment for executing side-effects generated by the LLM, such as shell commands or file operations.

## Key Features

- **Process Isolation**: Every agent session is mapped to a standalone worker process.
- **REST Interface**: Communication between Core and Sandbox happens over local HTTP to provide a logical firewall.
- **Resource Monitoring**: Tracks worker memory and CPU usage via `psutil`.
- **Graceful Lifecycle**: Automatic startup and shutdown of workers based on session activity.

## Boundary & Dependencies

### Inbound

- **Lane Queue / Runner**: Dispatches tool calls to the Sandbox Manager.

### Outbound

- **Host OS**: Executes shell commands and file operations within restricted paths.
- **Skill APIs**: The Sandbox worker loads and executes the actual tool logic.

## Extensions

- **Docker Driver**: Propose replacing subprocess isolation with short-lived Docker containers for higher security.
- **Capability Profiles**: Restrict workers to specific tool subsets based on the agent's "Role".

## Important Invariants

- **Port Management**: The Manager must ensure no port collisions between session workers.
- **Orphan Cleanup**: Manager must reap dead worker processes periodically.
