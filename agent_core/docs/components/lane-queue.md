# Component: Lane-Based Queue (`lane_queue/`)

## Responsibility

The `lane_queue` provides a reliable, asynchronous way to execute tool actions. It decouples the reasoning loop from the potentially slow or blocking tool execution, ensuring that the agent remains responsive.

## Core Concepts

### Lanes

A "Lane" is a logical partition of the command queue, typically associated with a single agent session.

- **Ordered Execution**: Within a single lane, commands are executed in strict FIFO order.
- **Concurrency**: Multiple lanes can be processed in parallel by different workers.

### Command Lifecycle

1. **Enqueued**: A tool call is generated by the agent and added to the queue via the `agent_core/loop.py`.
2. **Pending**: The command is waiting for an available worker assigned to its lane.
3. **Running**: A worker has picked up the command and is executing it within a sandbox.
4. **Completed / Failed**: The execution is finished, and the observation (or error) is recorded back to the `thoughts` table.

## Key Modules

- **`queue_engine.py`**: The primary interface for enqueuing and dequeuing commands.
- **`lane_manager.py`**: Manages the lifecycle and assignment of workers to specific lanes.
- **`persistence.py`**: Handles interaction with the Postgres database for command storage and status updates.

## Extending the Queue

New queue providers (e.g., Redis, RabbitMQ) can be implemented by adhering to the `BaseQueue` interface defined in `interfaces.py`.
